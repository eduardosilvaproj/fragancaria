RESUMO EXECUTIVO - MIGRAÇÃO FRAGRANCIARIA LOVABLE → RAILWAY
1. PROJETO E CONTEXTO
O que é:

Fragranciaria: e-commerce de produtos capilares (React + TypeScript)
Originalmente desenvolvido em Lovable.dev (plataforma proprietária)
Tech stack: TanStack Start + Vite + Nitro (não é Next.js)
Integrações: Mercado Pago, Supabase, Radix UI components
Por que migrar:

Dependência de plataforma fechada (Lovable)
Custos crescentes
Necessidade de independência e controle total
Deploy em infraestrutura própria/aberta
2. DESAFIOS ENCONTRADOS
Problema 1: Lovable Hardcoding
O projeto usava @lovable.dev/vite-tanstack-config que:

Auto-injetava 10+ plugins Vite (Nitro, React, Tailwind, etc)
Hardcodava preset Nitro como cloudflare-module (não ideal para Node.js)
Impossível remover sem quebrar build
Solução: Substituir por vite.config.ts customizado com Nitro para Node.js

Problema 2: Build Determinístico
Build local gerava hashes diferentes do build em CI/Railway
Exemplo: styles-CXo_rpW6.css (local) vs styles-DTLzBJFA.css (Railway)
Causa: Vite usa hash content-based; timestamps/env vars mudam entre ambientes
Solução: Dockerfile single-stage (eliminou problemas de cache entre estágios)

Problema 3: Assets não servidos
start.js só servia /assets/ de dist/client/
Faltava: Assets de /dist/server/ (bundles server-side do Nitro)
Faltava: Imagens de /images/
Resultado: Página carregava mas CSS, JS e imagens retornavam 404
Solução: Modificar start.js para servir de múltiplas localizações

5. FLUXO DE REQUEST
Request HTTP
    ↓
start.js (port 8080)
    ↓
├─ /assets/* → serve dist/server/assets/ ou dist/client/assets/
├─ /images/* → serve dist/client/images/
└─ Qualquer outra → Nitro handler
    ↓
server.js (Nitro entry)
    ↓
React Router SSR
    ↓
HTML + CSS + JS inlined
    ↓
Client hydration no browser
6. CONFIGURAÇÕES CRÍTICAS
vite.config.ts
defineConfig({
  tanstackStart: {
    server: { entry: "server" }  // SSR entry
  },
  vite: {
    build: {
      manualChunks: {
        'products-data': src/data/products,
        'vendor-framer': framer-motion,
        'vendor-icons': lucide-react
      }
    }
  }
})
start.js (Static Server)
// Serve múltiplas localizações
if (req.url.startsWith('/assets/')) {
  try dist/server/assets/ → dist/client/assets/
}
if (req.url.startsWith('/images/')) {
  try dist/client/images/
}
// Rest → Nitro SSR handler
Dockerfile (Single-Stage)
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install && npm run build
EXPOSE 8080
CMD ["node", "start.js"]
7. PROCESSO DE BUILD
npm run build
O que acontece:

Vite analisa src/routes/__root.tsx (entrada)
Gera bundle client:
dist/client/assets/index-{HASH}.js (main)
dist/client/assets/vendor-{HASH}.js (chunks)
dist/client/assets/styles-{HASH}.css (Tailwind)
Copia src/images/ → dist/client/images/
Gera bundle server (Nitro):
dist/server/assets/server-{HASH}.js (SSR handler)
dist/server/assets/*.js (route chunks)
Manifesto de rotas em _tanstack-start-manifest_*.js
Node script pós-build:
Cria dist/server/index.js (entry point para Railway)
